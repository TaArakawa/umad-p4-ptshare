# CLAUDE.md

このファイルは、本リポジトリで作業する Claude Code（claude.ai/code）向けのガイダンスを提供する。

## このプロジェクトの概要

FFXIVの高難易度レイド「絶妖星乱舞」（P1〜P5）用の、日本語の「カンペ（チートシート）」Webアプリ。ビルドステップ・フレームワーク・package.json を持たない静的サイト。パーティメンバーが戦闘中に開き、ギミックごとに真（true）／偽（false）どちらで解決したか、そしてそれが意味する行動（頭割り／散開、見る／見ない、動く／止まる 等）を確認する。ボスの真偽解決の状態は Firebase Realtime Database を介して接続中の全スマホ・PCへライブ同期され、1人の入力が全員の画面に反映される。

P4フェーズには実行時に切り替え可能な3つの独立実装がある:
- **v1** — `index.html` + `app.js`。最初の単一ページ・スクロール実装。
- **v2** — `solver.html` + `solver.js`（ESモジュール）。v1をテーブルベースで再設計したもの。
- **v3** — `solver-v2.html` + `solver-v2.js`（非モジュール、`firebase-*-compat.js` を使用）。現行かつ最も活発に開発しているバージョン。`hub.html` 内にも直接埋め込まれている。

ユーザーが実際に開くエントリポイントは `hub.html`。P1/P2/P3/P5のタブ（それぞれ独立したHTML/CSSパネルで、専用JSファイルは持たない）に加え、v1/v2/v3のどれを表示するか選べるP4タブを持つ。v1・v2は `<iframe data-src="index.html">` / `<iframe data-src="solver.html">` で埋め込まれる（DOMのid構造が hub.html 自身のマークアップと異なるため、iframeで隔離する必要がある）。**v3は hub.html 自身のDOMへ直接・無改変で埋め込まれ**、単独の `solver-v2.html` を動かすのと同じ `solver-v2.js` によって駆動される。したがって `solver-v2.js` は、hub.html のインラインP4マークアップと solver-v2.html のマークアップの両方に対して同時に動作し続けなければならない。

全ページが読み込む共有スクリプト:
- `theme.js` — ライト/ダーク背景の切り替え。唯一の真実は `localStorage['kfk_theme']` で、250msごとにポーリングして iframe と親のハブを同期させる。
- `nav.js` — P1〜P5のフェーズタブをタブ間・デバイス間で同期する（独自のFirebaseアプリインスタンス名 `"nav"` を使用。app.js/solver*.js が使うデフォルトアプリとは別にし、同一ページで両スクリプトが動いた際の二重初期化エラーを避けるため）。

## アーキテクチャ：共有のボス状態 vs. ローカルのプレイヤー状態

各P4実装（app.js / solver.js / solver-v2.js）は状態を2種類に分ける。ギミックロジックを編集する前に理解すべき最重要ポイント:

- **`bossState`** — ギミックごとのボスの実際の真偽解決（例: `gc1_truth`, `fire_truth`）を、安定した形に正規化したもの。**全クライアントで共有される**。app.js/solver.js（v1/v2）では Firebase v9 モジュラーSDK（`set(dbRef, currentState)` / `onValue(dbRef, ...)`）で `kfk_p4_state` に対し直接読み書きする。solver-v2.js（v3）では代わりに **compat SDK**（`dbRef.set(currentState)` / `dbRef.on('value', ...)`、727行目/521行目付近）を経由し、また `bossState` そのものが直接Firebaseに書かれるわけではない（下記の `currentState` の項を参照）。誰が入力しても全員に見える。
- **`currentState`**（v3/solver-v2.js のみ） — Firebase の `kfk_p4_state` に対して実際に読み書きされるワイヤーフォーマットのオブジェクト。v1のスキーマとの後方互換のため、古い/レガシーなキー形状（`earlyWater`, `fire`, `water`, `lineLightning`, `iceFan`, `gc1Truth`, ...）を使う。`applyFirebaseData()`（436行目付近）が受信した `currentState` から `bossState` の正規化フィールドを導出し、`updateFirebaseState()`（646行目付近）が書き込み前に逆方向のマッピングを行う。オフライン時には（`localState` ではなく）`currentState` が `localStorage['kfk_solver_offline_state']` にキャッシュされ、`storage` イベントでタブ間再同期される。このキーは特定プレイヤーのデバフとは無関係。
- **`localState`** — *特定のプレイヤー個人*が持つデバフ（例: `gc1_water_lightning: 'water'|'lightning'|null`, `gc1_sight`, `gc1_bomb`）。**端末ごとのみ**で、`localStorage['kfk_solver_local_state']` に永続化され、Firebaseへは決して送られない。パーティメンバーはそれぞれ同時に異なるデバフを持つため。

各 solver*.js の `renderUI()` は両者を組み合わせる。まず `localState` を見てどのデバフ行がアクティブ／選択可能かを決め（行同士の相互排他・クロス排他ルールを伴う。例: 視線(Sight)と加速度(Bomb)は同一のグランドクロス発生に紐づき、片方のGCに加速度がある場合はもう片方のGCの視線が排除される）、次に `bossState` を見てどの真偽セルを解決行動としてハイライトするかを決める。

新しいデバフ／ギミック制約を追加する際のパターンは以下の通り。`localState` にフィールドを追加し（`loadLocalState()`/`saveLocalState()` の処理を伴う）、`renderUI()` に排他ロジックを追加し（行要素の `active`/`inactive`/`sight-selected` 系クラスを切り替える）、クリックハンドラをガードして `inactive` 行のクリックをno-opにする。UIのクラスとクリックガードは必ず一致させること。ずれると「押せそうに見えるのに何も起きない」行（あるいはその逆）が生まれる。

Firebaseの設定（apiKey 等）は、オープンなルールの Firebase Realtime Database 向けの、意図的に公開されたクライアント設定。app.js/solver.js/solver-v2.js/nav.js に一字一句そのまま複製されている。変更する場合は全ファイルで同期を保つこと。

## 開発ワークフロー

ビルド/lint/testのツールは存在しない（package.json も テストファイルもない）。アプリの「実行」とは、HTMLを直接開くか、ディレクトリを静的配信することを指す。例:

```bash
python3 -m http.server 8000   # その後 http://localhost:8000/hub.html を開く
```

バンドラは無い。変更の検証はコンパイル/型チェックではなく、ブラウザ（または Claude Preview ツール）でページを読み込んでUIを実際に操作して行う。

### キャッシュバスティングの規約

HTMLファイル内のすべての `<script src="...">` / `<link href="...">` は `?v=YYYYMMDDx` のクエリ文字列を使う（例: `solver.js?v=20260701a`）。**共有のJS/CSSファイルを編集したら、それを参照している全HTMLファイルで `?v=` を上げること**。さもないとブラウザ/CDNが古いキャッシュを返す可能性がある。v3の `solver-v2.js` は `solver-v2.html` と `hub.html` の両方から参照されるため、両方まとめて上げること。

### バージョン更新

`version.json`（`{"version": "x.y.z"}`）は、ユーザーから見える変更を出荷するコミットごとに、`?v=` のキャッシュバスティングとは独立して、きっかり `+0.0.1` 上げる。上げる前に `git log -p -- version.json` で現在のパターンを確認すること。

### Gitワークフロー

コミットは Conventional Commits スタイル（`feat: ...`, `fix: ...`）を使い、変更が特定の実装のみに触れる場合は `(v2/v3)` のような短いスコープ注記を付ける。各変更はそれ専用の `feature/*` または `fix/*` ブランチで出荷し、`main` へのPRとして開き、（squash/rebaseではなく）マージコミットでマージする。1つの焦点を絞った変更につき1PR、という確立したパターンは `git log --oneline` を参照。同一セッションで開発した場合でも、無関係な変更（例: 表示/テーマ作業 vs. ギミックロジック作業）は別々のコミット/ブランチに分けること。

## タスク委譲ルール（Antigravity CLI）

### 役割分担の大原則

Claude（自分）が担うのは、以下の3つの「判断」に限る。それ以外の実作業（箇所の特定を含む）はできるだけ agy に委譲すること：

1. **指示内容の分解** — ユーザーの依頼を、委譲可能なサブタスクへ切り分ける。並列化できるか（後述）、および各サブタスクを2段階委譲と1段階委譲のどちらで処理するか（後述）もここで判断する。
2. **agy の特定結果の検証** — 「どこを、何を、なぜ直す（調べる）か」の特定作業そのものは agy に委譲する。ただし2段階委譲では、agy が返してきた特定内容をそのまま実装させず、一度受け取って正しそうか（対象箇所・原因・修正方針が妥当か）を Claude が判断する。妥当でなければ差し戻す。
3. **agy の処理結果のレビュー** — 実装後に返ってきた差分・調査結果を検証し、採否を決める。

つまり Claude は「分解」「特定結果の検証（実装前ゲート）」「実装結果のレビュー」という判断ポイントを担い、実作業（箇所の特定・調査・コード実装・生成・リファクタ・網羅的な読み取り／検索／要約）は原則すべて agy に委譲する。自分の Grep/Read/Edit/Bash ツールで実作業を完結させてはならない。

### 2段階委譲と1段階委譲の使い分け

サブタスクごとに、変更の影響範囲とリスクに応じて以下を使い分ける。この判断は「分解」の段階で行う。2段階委譲は検証ゲートのぶん Claude 側のトークンを追加で消費するため、保険が必要な場面に絞って使う。

**2段階委譲（特定 → 検証ゲート → 実装）を使う** — 影響範囲が広い、または壊れたときの影響が大きい変更。誤った特定のまま実装が走るのを防ぐため、実装前に Claude の検証ゲートを挟む。該当する例：
- 状態管理ロジック（bossState / currentState / localState、renderUI の排他ロジック等）に触れる変更
- Firebase 同期まわり（dbRef の読み書き、applyFirebaseData / updateFirebaseState 等）に触れる変更
- 複数ファイル横断で影響し合う変更、または v1/v2/v3 の複数実装にまたがる変更
- キャッシュバスティングやバージョン更新など、規約違反が本番の不具合に直結する箇所

**1段階委譲（特定と実装をまとめて agy に一括）を使う** — 軽微・自明で、壊れても影響が小さい変更。検証ゲートのオーバーヘッド（Claude 側の追加トークン消費）を省いて効率を優先する。該当する例：
- 表記ゆれの統一、コメント・文言の修正
- 単一ファイル内で完結し、他の実装や状態管理に波及しない局所的な修正
- ロジックを変えない純粋な整形・リネーム

判断に迷う場合は、安全側に倒して2段階委譲を選ぶ。

### 標準の2段階委譲フロー

2段階委譲を選んだサブタスクは、以下の順で処理する：

1. **[agyに委譲] 特定** — 「どのファイルの、どこを、何のために変更（調査）すべきか」を agy に調べさせる。この段階では実装・変更はさせず、特定結果（対象箇所・原因・修正方針）だけを報告させる。
2. **[Claudeが判断] 検証ゲート** — 返ってきた特定内容が正しそうかを Claude が検証する。妥当なら次へ進め、疑わしければ追加調査や再特定を agy に指示する。原因が自明に思えても、この検証を省いて自分で実装を書き始めてはならない。
3. **[agyに委譲] 実装** — 検証を通過した特定内容に基づき、実際のコード変更・生成・リファクタを agy に実行させる。
4. **[Claudeが判断] レビュー** — 返ってきた差分を Claude がレビューし、採否を決める。

1段階委譲を選んだサブタスクは、上記の1〜2を省略し、特定と実装をまとめて agy に依頼したうえで、返ってきた結果を Claude がレビュー（上記4）する。

### 委譲対象（該当すれば明示指示がなくても委譲する）

以下は上記フローに乗せて必ず agy に委譲すること。自分の Grep/Read/Bash ツールで直接完結させてはならない：
- 既存コードの箇所特定・調査・要約
- 定型的な実装・テストコード生成
- 単純なリファクタ

「調査して」と明示されていない、"はみ出てる/崩れてる/動かないから直して" のような修正依頼でも、上記に該当する部分は同様に委譲対象。

### 「バグを直して」系タスクでのフェーズ分け

特にブラウザでの再現・検証を伴うUI/表示バグ修正では、以下のようにフェーズを分けること。ここでは agy に委譲できない工程（ブラウザ実測）を Claude が担い、それ以外を agy に回す。原則2段階委譲だが、症状が自明かつ局所的な場合は上記の使い分けに従い特定と実装をまとめてよい：

1. **自分で行う（agyにはできない部分）**: preview_* ツールでの再現、スクリーンショット、`preview_inspect` による崩れている要素の実測値（セレクタ・幅・フォントサイズ・テキスト内容等）の取得。
2. **agyに委譲する（特定）**: 1で得た具体的な症状（実測値・セレクタ・ファイル名の見当）を渡した上で、「原因となっているCSS/コード箇所の特定」と「修正方針」を報告させる（この時点では実装させない）。
3. **自分で判断する（検証ゲート）**: agy の特定した原因・方針が妥当かをレビューする。
4. **agyに委譲する（実装）**: 検証を通過した方針で修正を実装させる。
5. **自分で行う**: agy の差分のレビュー、preview_* ツールでの再検証、コミット/PR/マージ。

原因が自明に思えても、コード箇所の特定と修正実装をその場で自分で完結させてはならない。症状を再現・実測できた時点で一度立ち止まり、委譲すべきサブタスクが含まれていないか確認する。

### 並列実行のルール

複数の委譲対象サブタスクが存在する場合、以下の両方を満たすときは並列で実行すること：
- 対象となるファイル・ディレクトリが互いに重複しない（同じファイルを複数のサブタスクが同時に編集しない）
- サブタスク間に依存関係がない（一方の結果を待たないと他方を開始できない、という関係にない）

いずれかを満たさない場合（同一ファイルへの複数書き込みが発生する、あるいは後続タスクが先行タスクの結果を必要とする）は、無理に並列化せず逐次実行すること。

並列実行する際は、各サブタスクにタスクIDを割り当て、出力を `/tmp/agy-result-<task-id>.txt` のように分離すること。ワークスペースが重複しない場合でも、可能な範囲で git worktree 等によりさらに作業領域を隔離することが望ましい。

なお、上記の2段階委譲フロー（特定→検証→実装）は各サブタスク内でもそのまま適用する。並列化した場合、各サブタスクの「特定」段階を並列で走らせ、それぞれの検証ゲートを通過したものから「実装」段階へ進める（1段階委譲を選んだサブタスクはそのまま並列で実装まで走らせてよい）。

### レビューの並列化と統合レビュー

並列実行した複数の agy 委譲結果は、それぞれが単体で完結して検証できる場合（他のサブタスクの結果と突き合わせる必要がない場合）は、レビューも並列に行ってよい。

個別レビューがすべて完了した時点で、必ず全体を俯瞰した統合レビューを行うこと。統合レビューでは特に以下を確認する：
- 各サブタスクの変更が互いに矛盾していないか（命名規則、状態管理の整合性、既存のアーキテクチャ規約との整合性等）
- 全体として当初の目的を達成しているか
- CLAUDE.md の既存ルール（bossState/currentState/localState の分離、キャッシュバスティング規約、バージョン更新規約等）に違反していないか

個別レビューが1件でも未完了のうちは、統合レビューを開始しないこと。

### 委譲時の共通ルール

- delegate-to-agy.sh の出力は必ずレビューしてから採用すること
- 空出力やエラーが返った場合はリトライせず自分で処理を引き取ること
- 機密情報・認証情報を含むプロンプトは絶対に渡さないこと
- 作業ディレクトリは必ず絶対パスで指定すること（相対パスは正しく解決されない）
